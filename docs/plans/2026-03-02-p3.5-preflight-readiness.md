# P3.5 前置就绪清单（Preflight）

日期：2026-03-02  
状态：可进入执行排期（尚未实施 P3.5 代码改造）

## 1. 目标与边界

本文件用于在执行 `docs/plans/2026-03-02-p3.5-acp-implementation.md` 前，明确当前代码库与目标态的差距、分批改造顺序、风险与门禁。

本次结论基于并行审计（配置层 / Pipeline Action 链路 / Secretary-Review 链路），不包含业务逻辑改造提交。

## 2. 结论摘要

当前仓库尚未进入 P3.5 目标态。主要差距：

1. 配置层仍使用 `secretary.review_panel`，目标应为 `secretary.review_orchestrator`。
2. Pipeline Action 仍使用 `change_agent + agent`，目标应为 `change_role + role`。
3. Secretary 审核编排仍为 `ReviewPanel` 命名与实现，目标应对齐 `ReviewOrchestrator`（并最终进入 role-driven ACP 调用）。
4. 文档（spec/plans）与代码口径已出现漂移，若不先收口会导致执行阶段反复返工。

## 4. 推荐执行顺序（供执行团队）

### Phase A：契约与命名骨架（低风险收口）

1. 配置字段命名骨架：`review_panel -> review_orchestrator`
2. 核心动作命名骨架：`change_agent -> change_role`、`agent -> role`
3. Secretary 命名骨架：`ReviewPanel -> ReviewOrchestrator`

交付标准：

1. 编译通过。
2. 旧术语只保留在必要兼容分支或迁移注释中。

### Phase B：入口链路切换（Web/CLI/TUI/API）

1. `handlers_pipeline`、CLI、TUI、前端 API 类型同步改为 `change_role + role`。
2. 事件透传字段切换到 `role`（如需，短期双写）。

交付标准：

1. API / CLI / TUI 文案一致。
2. Web 请求不会因字段口径漂移触发大量 `INVALID_JSON`。

### Phase C：行为落地（Role-driven ACP）

1. Stage 执行按 `stage.role`（或等价映射）驱动。
2. Review Orchestrator 接入 `role_bindings.review_orchestrator`。
3. Secretary、Plan Parser、Pipeline 全部通过 role 解析调用 ACP。

交付标准：

1. 与 `p3.5` wave2/wave3 验收一致。
2. 关键路径测试补齐。

## 5. 风险与防护

高风险项：

1. 配置键改名导致历史配置静默失效。
2. 文档已改、代码未改导致联调反复失败。
3. 审核编排“只改名不改行为”造成验收假通过。

防护策略：

1. 每个 phase 结束执行一次全局术语扫描。
2. 对关键动作链路建立专项测试（`change_role` 正反向）。
3. 明确回退策略：`LoadSession -> NewSession` 与 ReviewGate fallback。

## 6. 执行门禁（建议）

执行团队进入 P3.5 前，先确认：

1. 迁移是否允许破坏性（当前口径：允许）。
2. 是否保留旧配置兼容窗口（需最终拍板）。
3. 术语门禁命令是否排除 archive 文档（避免误报）。

推荐命令：

```bash
rg -n 'review_panel|change_agent|implement_agent' internal cmd configs web docs/spec -g '!docs/plans/archive/**'
rg -n 'review_orchestrator|change_role|implement_role|stage\.role' docs/spec docs/plans internal cmd configs web
go build ./...
```
