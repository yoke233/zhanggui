# ACP Role-Driven Implementation Plan

> **For Agent:** REQUIRED SUB-SKILL: Use executing-wave-plans to implement this plan wave-by-wave.

**Goal:** 交付 ACP-only、Role-driven 的统一 Agent 调用体系：以 `agents + roles + role_bindings` 作为唯一配置模型，完成 Executor / Secretary / Review Orchestrator / Plan Parser 的角色化接入，并完成 Workspace 收口。

**Architecture:** 按“基础设施 → 执行链路 → 交互链路 → 工程收口”推进。Wave 1 建立生产级 `internal/acpclient/` 与 RoleResolver。Wave 2 让 Pipeline/Plan Parser 全量按 role 调用。Wave 3 让 Secretary/Review Orchestrator 全量按 role 调用并默认复用会话。Wave 4 完成冗余通信层清理、Workspace 插件收口与 `remote_url` 统一模式。

**Tech Stack:** Go 1.25+, chi v5, SQLite (modernc.org), gorilla/websocket, React + Tailwind + Vite, ACP (JSON-RPC 2.0 over stdio)

---

## 1. Context and Scope

### In Scope

- `internal/acpclient/` 生产级包：transport、protocol、handler、client、测试
- RoleResolver：`role_id -> AgentProfile + RoleProfile` 解析与约束校验
- 配置模型落地：`agents.capabilities_max` + `roles` + `role_bindings`
- Executor (`internal/engine/executor.go`) 改为按 `stage.role` 执行 ACP
- Plan Parser 改为按 `role_bindings.plan_parser.role` 执行 ACP
- Secretary (`internal/secretary/agent.go`) 改为按 `role_bindings.secretary.role` 持久会话
- Review Orchestrator 改为按角色调用 ACP（reviewer + aggregator 默认复用）
- API 动作统一为 `change_role`（角色切换语义）
- Workspace：worktree 插件独立 + clone `remote_url` 单输入模式
- Factory 与 BootstrapSet 适配（启动即构建 `agents + roles + role_bindings`）

### Out of Scope

- P4 功能：自定义模板扩展、Notifier 多通道增强、Reactions V2、治理增强
- 改写 Pipeline 核心状态机语义
- 分布式调度 / 多实例部署
- 前端框架更换

### 关键约束

- `executing-wave-plans` 波次门禁：未达 `Go` 不得进入下一波
- 单一路径：所有 Agent 调用统一走 ACP + role 解析
- 配置严格 schema 校验：未知字段/非法枚举/类型不匹配直接 fail-fast
- 默认会话策略：Pipeline `hybrid`；Review Orchestrator 的 `reviewer` 与 `aggregator` 角色默认 `reuse=true`

## 4. Global Quality Gates (F/Q/C/D)

### F - Functional
- [ ] `internal/acpclient/` 提供 `New/Initialize/NewSession/LoadSession/Prompt/Cancel/Close` 完整 API。
- [ ] Handler 覆盖 `read/write file`、`permission`、`terminal/*` 全部回调。
- [ ] RoleResolver 可稳定完成 `role_id -> AgentProfile + RoleProfile` 解析。
- [ ] Pipeline Stage 完整执行链路为 `AcquireSession -> Prompt -> end_turn -> ReleaseSession`，按 role 策略复用。
- [ ] Secretary 多轮会话按 `secretary role` 运行，支持 `LoadSession -> NewSession`。
- [ ] Review Orchestrator（三个 reviewer + aggregator）全部按 role 调用，默认复用会话。
- [ ] Plan Parser 按 `plan_parser role` 一次性调用完成解析。
- [ ] 文件变更由 `HandleWriteFile` 作为唯一事实来源。
- [ ] Workspace：worktree 插件可用，clone 支持 `remote_url` 单输入。

### Q - Quality
- [ ] 新增包单测覆盖核心路径 >= 80%。
- [ ] 每个 Wave 至少 1 条 wave 级 smoke/e2e 通过。
- [ ] `go vet ./...` 无新增警告。

### C - Configuration
- [ ] 全局/项目配置仅使用 `agents + roles + role_bindings + pipeline + secretary`。
- [ ] 配置解析严格 fail-fast（未知字段、类型错误、非法枚举）。
- [ ] API 与文档统一使用 `change_role`、`implement_role`、`stage.role`、`review_orchestrator` 口径。

### D - Documentation
- [ ] `docs/spec/` 与代码口径一致。
- [ ] Role-driven 配置样例可直接执行。

## 8. Test Policy

- 每个任务遵循 TDD：先失败测试 → 最小实现 → 回归。
- 每个 Wave 必须包含至少 1 条 wave 级 smoke/e2e。
- ACP 依赖测试使用 fake subprocess / mock transport。
- `internal/acpclient/` 必须有独立 transport mock，不依赖真实 Agent 进程。

## 9. Assumptions

- ACP-compatible Agent 二进制可稳定运行（至少一个可用）。
- 配置模型以 `agents + roles + role_bindings` 为唯一来源。
- `remote_url` 单输入支持主流 Git 远程地址（SSH/HTTPS）。
- P4 功能在本计划完成后独立规划。

## 10. Execution Handoff

- 当前会话：按 Wave 1 → Wave 4 顺序执行，每波输出门禁证据。
- 并行会话：仅允许在同一 wave 内无冲突文件任务并行。
- 任意波次触发高风险回归时，回到上一波通过基线再推进。
- Wave 4 Gate 通过后，进入 P4 独立计划。
