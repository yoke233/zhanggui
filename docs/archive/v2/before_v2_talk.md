  └────────────┴────────────────────────────────────────┴────────────────────────────────────────┘

  ---
  设计演进：三个关键决策

  决策 1：Submission 取代 Artifact

  ARCHITECTURE.md 提出的 result.md + assets/ 统一交付模型是对的。v2 的 Artifact（7 种 kind）过度分类且无法验证。

  演进方案：

  Submission
  ├── result.md          — 唯一主产物，结构化 Markdown
  ├── assets/            — 附件
  └── metadata           — 机器可读的结构化摘要（从 result.md 提取）

  关键点：result.md 是人写的/AI 写的自然语言，metadata 是引擎提取的结构化数据。这样：
  - Gate agent 读 result.md 做审核（自然语言层）
  - 引擎读 metadata 做自动校验（机器层）
  - 下游 step 两层都能用

  metadata 不需要预定义 schema，只需要几个约定字段：

  # metadata 约定字段（从 result.md 首部 frontmatter 提取）
  status: completed | partial | blocked
  deliverables:         # 交付清单
    - type: branch
      ref: "feat/login"
    - type: file
      path: "internal/auth/handler.go"
  issues: []            # 发现的问题
  next_steps: []        # 给下游的建议

  这就把 v2 的 7 种 Artifact kind 和 ARCHITECTURE.md 的 Submission 统一了。Markdown 是主体，结构化是副产品。

  决策 2：Step Briefing 取代 PromptTemplate + InputSchema

  v1 用模板渲染 prompt（硬编码变量槽位），v2 完全没有 prompt 层，ARCHITECTURE.md 说"平台不负责规定具体提示词细节"。

  三者都不完全对。正确的分层是：

  引擎负责的（Briefing）          Agent 负责的（Prompt 细节）
  ─────────────────────          ──────────────────────
  你要做什么（目标）               怎么思考
  你能看到什么（上下文引用）        用什么策略
  你能做什么（能力边界）           调用什么工具
  做完怎么交（Submission 格式）     具体提示词措辞
  什么算合格（验收标准）

  Briefing 是引擎给 Agent 的"任务书"，不是 prompt。Agent 拿到 Briefing 后自己决定怎么构造 prompt。

  Step.Briefing:
    objective: "实现用户登录接口"
    context_refs:                    # 引擎自动解析，注入可访问的上下文
      - flow_summary                 # 当前 Flow 摘要
      - upstream_submissions: [3, 5] # 上游 step 3, 5 的 Submission
      - project_brief                # 项目简介
    acceptance_criteria:             # Gate 评判标准
      - "API 通过 smoke test"
      - "代码有单元测试"
    capabilities_required:           # 需要的能力
      - "dev.backend"

  引擎的职责是：组装 Briefing → 校验 Agent 是否有对应 capability → 执行 → 收 Submission → 校验 metadata。不碰 prompt。

  这比 v1 的 PromptBuilder（侵入 prompt 细节）更干净，比 v2 的裸 map[string]any（啥也没有）更有约束力。

  决策 3：能力模型 = Tag 匹配 + Action 白名单

  v1 的 3-bool 模型（fs_read/fs_write/terminal）太粗；ARCHITECTURE.md 的 capability tag 太抽象。结合两者：

  两层能力：

  1. Capability Tags（业务能力）—— 决定 agent 能不能接这个 step
     agent: ["dev.backend", "dev.frontend"]
     step:  requires ["dev.backend"]
     → 匹配：可以接

  2. Action Whitelist（执行动作）—— 决定 agent 在执行中能做什么
     role=worker:  [read_context, search_files, submit_submission, mark_blocked]
     role=gate:    [read_context, search_files, submit_submission, approve, reject]
     role=lead:    [read_context, search_files, submit_submission, create_step, expand_flow]

  关键区别：
  - Capability Tag 是"你擅长什么"（选人）
  - Action Whitelist 是"你被允许做什么"（管行为）

  v1 的 fs_read/fs_write/terminal 其实是 Action 层的东西，不应该和业务能力混在一起。

  实际的 Action 清单（精简自 ARCHITECTURE.md 的 5 个 + v1 的 3 个）：

  ┌───────────────────┬──────┬────────┬──────┬─────────┐

  ├───────────────────┼──────┼────────┼──────┼─────────┤
  │ read_context      │ Y    │ Y      │ Y    │ Y       │
  ├───────────────────┼──────┼────────┼──────┼─────────┤
  │ search_files      │ Y    │ Y      │ Y    │ Y       │
  ├───────────────────┼──────┼────────┼──────┼─────────┤
  │ fs_write          │ Y    │ Y      │ N    │ N       │
  ├───────────────────┼──────┼────────┼──────┼─────────┤
  │ terminal          │ Y    │ Y      │ N    │ N       │
  ├───────────────────┼──────┼────────┼──────┼─────────┤
  │ submit_submission │ Y    │ Y      │ Y    │ Y       │
  ├───────────────────┼──────┼────────┼──────┼─────────┤
  │ mark_blocked      │ Y    │ Y      │ Y    │ Y       │

  ---
  引擎执行管道

  把 v2 的单一 StepExecutor func 拆成管道：

  1. Resolve     选人：Step.RequiredCapabilities → 匹配 AgentProfile
  2. Brief       组卷：组装 Briefing（注入上游 Submission、Project 摘要等）
  3. Guard       校验：Agent 的 Action Whitelist 是否覆盖 Step 需要的操作
  4. Execute     执行：启动 Agent，传入 Briefing，监听 Action 调用
  5. Collect     收件：接收 Submission，提取 metadata
  6. Validate    验收：自动检查（metadata.status、必填字段）
  7. Transition  流转：更新 Step 状态，推动下游

  每一步都是引擎控制的，Agent 只参与第 4 步。这就是"约束层"——不是管 Agent 怎么想，而是管 Agent
  的输入是什么、能做什么、输出必须是什么。

  对比：
  - v2 现在：只有 4 和 7（执行 + 流转）
  - v1：有 1、2、3、4、7（但 2 侵入了 prompt 细节）
  - 演进后：7 步完整，且每步职责清晰

  ---
  Gate 集成方案

  ARCHITECTURE.md 说"Gate 是一种 Step"，v2 代码也这么做了，但 ProcessGate 是外部手动触发的。演进方案：

  Gate Step 执行流程：
  1. 引擎自动收集上游 Submission（所有 DependsOn 的最新 Submission）
  2. 组装 Gate Briefing：
     - 待审物：上游 Submission 列表
     - 验收标准：上游 Step 的 AcceptanceCriteria
     - 审核指引：Gate Step 自己的 Config
  3. 分配 role=gate 的 Agent
  4. Agent 审核后提交 Submission：
     - metadata.verdict: pass | reject
     - metadata.reject_targets: [step_id, ...]  # reject 时指定打回哪些
     - result.md: 审核意见详情
  5. 引擎读 metadata.verdict：
     - pass → Gate done → 下游 promote
     - reject → 上游 reset → 带上审核意见作为下一次 Briefing 的输入

  这样 Gate 完全自动化，不需要外部手动调 ProcessGate。审核意见自然流入下一次 Execution 的 Briefing。

  ---
  Composite 集成方案

  Composite Step 执行流程：
  1. 引擎检测到 Step.Type == composite
  2. 分配 role=lead 的 Agent
  3. Agent 提交 Submission：
     - metadata.sub_steps: [{name, type, depends_on, capabilities_required}, ...]
  4. 引擎读 metadata.sub_steps → 创建子 Flow + 子 Steps
  5. 递归 Run 子 Flow
  6. 子 Flow done → 父 Step done
  7. 子 Flow failed → 父 Step 可重试（Lead 重新拆分）

  Lead 只负责拆分，引擎负责执行。拆分结果是结构化数据（在 Submission.metadata 里），不是 prompt。

  砍掉的: Message, Thread, Workspace, Pattern, Schedule, DashboardData, Agent 组织树

  关键区别：用 Submission 取代 Message + Artifact

  v3 的信息流：Agent A → Message → Bus → Agent B（异步消息驱动）

  v2 演进的信息流：Step A done → Submission 写入 → 引擎推导 Step B ready → 注入 Briefing（引擎驱动，Submission
  是数据载体）

  不需要消息系统。 Step 之间的数据传递通过引擎读上游 Submission → 组装下游 Briefing
  完成。这比消息投递简单一个数量级，且引擎对数据流有完全控制权。

  保留 v3 的 Prompt 质量理念

  Briefing 不是 prompt，是给 prompt 的素材。组装顺序遵循 v3 的 prefix cache 友好排列：

  1. Agent 身份（System Prompt）     ← 几乎不变，cache 命中
  2. Project 摘要                    ← 项目周期内稳定
  3. Flow 摘要 + 验收标准            ← Flow 周期内稳定
  4. 上游 Submission 摘要（温）      ← 偶尔变
  5. 当前 Step 的 Briefing（热）     ← 每次执行都变

  引擎组装 Briefing 时遵循这个顺序。Agent 拿到 Briefing 后自己决定怎么用——引擎提供素材，不侵入 prompt 细节。

  约束层：从 v1 拿回来、用 ARCHITECTURE.md 的方式组织

  Agent Profile:
    role: lead | worker | gate | support    ← ARCHITECTURE.md 四角色
    capabilities: ["dev.backend", ...]      ← ARCHITECTURE.md tag
    actions: [read_context, fs_write, ...]  ← v1 能力的演进
    driver: claude | codex | human          ← v1 agent 绑定

  Step 声明:
    required_capabilities: ["dev.backend"]  ← 选人
    allowed_actions: [fs_write, terminal]   ← 管行为（可选，默认用 role 的）
    acceptance_criteria: [...]              ← Gate 评判标准
    timeout: 30m                            ← 超时

  引擎执行管道（7 步）：
  Resolve → Brief → Guard → Execute → Collect → Validate → Transition

  每一步引擎控制，Agent 只参与 Execute。v3 的"控制面只当邮局"理念在 v2 里演化为"引擎管约束，Agent 管执行"。
