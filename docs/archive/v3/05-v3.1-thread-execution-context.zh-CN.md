## 1. 引入Thread（会话容器）

### 1.1 为什么需要Thread

v3将Session完全合并入Task，但遗留了三个问题：

**任务诞生前的闲聊没有归属地。** `Message.task_id`为空的消息是"孤儿"。

**同一个Task下的消息混为一团。** 需求澄清、技术争论、复盘总结都挂在同一个task_id下，PromptBuilder取热上下文时会把不相关的讨论也塞进去。

**两种事实被混在一起。** TaskStep是"状态事实"（task created, review rejected），Message是"认知事实"（某人提出担忧、两个agent辩论技术路线）。它们的稳定性完全不同。

### 1.2 Thread模型

```javascript
Thread
├── id
├── task_id             可为空（任务前闲聊）
├── participants        参与者列表
│                       有task_id时从Task.participants派生
│                       无task_id时必须显式指定
├── topic               AI自己填的话题摘要
├── status              open / crystallized / closed
└── created_at
```

### 1.3 Message变更

```javascript
Message
├── ...v3现有字段
├── thread_id           必填（每条消息必须属于一个Thread）
├── reply_to_msg_id     可选（回复哪条消息，群聊时用于追踪对话链）
├── task_id             可选（便于索引，从Thread.task_id派生）
```

`reply_to_msg_id`的作用：在群聊场景下，多个agent同时发言，话题可能分叉。没有reply_to，消息是一条扁平时间线，无法区分谁在回复谁。PromptBuilder可以沿reply_to链提取完整对话上下文，而不是简单取最近N条。

### 1.4 典型场景

**场景1：任务前闲聊**
- Thread{task_id: null, participants: [human, supervisor]}
- 到crystallize时创建Task
- Thread.status = crystallized
- Thread.task_id回填新task_id

**场景2：任务中的技术讨论**
- Task已存在
- 在task下新建Thread{topic: "JWT vs Session认证方案"}
- 讨论完成后产出artifact

**场景3：任务中的审阅**
- 单独一个Thread{topic: "代码审阅"}
- 审阅通过后才写TaskStep

**场景4：群聊**
- Thread{participants: [supervisor, dev_1, dev_2, test_1]}
- 多人自由讨论，用reply_to_msg_id追踪对话链
- 不需要正式的"开会"流程

## 2. Bus支持Thread投递（群聊）

### 2.1 变更

Bus的Send方法支持`msg.to`填thread_id。Bus检测到目标是thread时，查Thread.participants列表，广播给所有参与者（排除发送者自己）。

```go
func (b *Bus) Send(msg Message) error {
    if isThreadID(msg.To) {
        participants := b.store.GetThreadParticipants(msg.To)
        for _, p := range participants {
            if p != msg.From {
                b.deliverTo(p, msg)
            }
        }
    } else {
        b.deliverTo(msg.To, msg)
    }
}
```

### 2.2 单聊 vs 群聊

| 场景 | `msg.to` | 投递方式 |
|---|---|---|
| 单聊/派活/求助 | `agent_id` | 点对点 |
| 群聊/讨论 | `thread_id` | 广播给所有participants |

不需要区分"单聊"和"群聊"的消息类型。Bus根据 `msg.to` 的目标自动判断。

## 3. Workspace降级为ExecutionContext

### 3.1 为什么降级

Workspace作为独立核心实体存在几个问题：
- 语义不稳定：有时是执行沙箱，有时是资源集合，有时是上下文作用域
- 和Task、Memory在语义上有交叉
- 引入OpenViking后，resources引用会被viking URI吸收
- 它更像是"执行时视图"，而不是高层核心对象

### 3.2 变更

**删除独立的Workspace实体。** 合并为Task的嵌套结构：

```javascript
Task
├── ...现有字段
├── execution_context         可选（讨论型task不需要）
│   ├── resource_refs         [{kind, uri}]
│   ├── credential_scope      凭证范围
│   └── environment_hint      环境描述（自然语言）
```

**WorkspaceProvider接口改名为ResourceProvider：**

```go
type ResourceProvider interface {
    Provision(task Task) (ExecutionContext, error)
    Cleanup(taskID string) error
}
```

输入是Task，输出是填充好的execution_context。接口签名基本不变，但模型更干净。

## 4. PromptBuilder的改进

### 4.1 热上下文从Thread维度取

之前：取当前task下最近N条消息（所有讨论混在一起）

现在：取当前thread的最近N条消息（精准到具体讨论）

### 4.2 对话链提取

群聊场景下，有了reply_to_msg_id，PromptBuilder可以做对话链提取：

```javascript
从当前消息出发
  → 沿reply_to_msg_id往回追溯
  → 拿到完整的对话链
  → 作为热上下文的核心内容
  → 同一Thread内不在对话链上的消息作为补充
```

比简单取最近N条精准得多，尤其在多人并行讨论时。

## 5. 完整模型变更清单

### 新增模型

| 模型 | 用途 |
|---|---|
| Thread | 会话容器，消息归属地 |

### 修改模型

| 模型 | 变更 |
|---|---|
| Message | 新增thread_id（必填）、reply_to_msg_id（可选） |
| Task | execution_context嵌套结构替代独立Workspace |
| Bus | Send支持thread_id作为投递目标 |

### 删除模型

| 模型 | 原因 |
|---|---|
| Workspace | 降级为Task.execution_context |

### 接口变更

| 接口 | 变更 |
|---|---|
| WorkspaceProvider → ResourceProvider | 改名，输入改为Task |
| Bus.Send | 支持thread_id广播 |
| Store | 新增Thread的CRUD + GetThreadParticipants |

### 不变的部分

状态机、TaskStep事件溯源、Decision版本化、AgentError错误分类、DecisionValidator、Watchdog、Memory接口签名、Notifier——全部不变。

---

## 6. 设计原则确认

| 决策 | 结论 | 理由 |
|---|---|---|
| Thread.kind要不要枚举 | 不枚举 | 保持自然语言描述，和其他语义字段一致 |
| reply_to_msg_id要不要加 | 加 | 群聊场景必须追踪对话链，PromptBuilder需要 |
| parent_thread_id要不要 | 不加 | 暂时不需要Thread嵌套，以后需要再加 |
| Workspace要不要保留 | 降级 | 合并为Task.execution_context，不再独立实体 |
| Bus要不要支持广播 | 支持 | `msg.to` 填thread_id时自动广播，不需要新的消息类型 |
