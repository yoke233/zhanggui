# v3 交付物落地设计讨论 — Artifact、代码提交、文件上传与存储

> 说明：本页整理 v3 设计讨论中已经确认的有效结论，以及仍需继续定稿的开放点。不记录已经放弃的方案。
>
> 时间：2026-03-09

## 1. 目标

围绕 v3 的 `Task / Thread / Artifact` 模型，回答一个落地问题：

**AI 完成任务后，怎么稳定地告诉系统“我交付了什么”，以及系统如何把这些交付物放到合适的位置。**

这里既包括代码类结果，也包括非代码类结果：

- 代码修改、分支、Pull Request
- Markdown 文档、方案稿、报告
- PPT、PDF、图片、视频
- 多文件打包结果

核心要求有两条：

- **AI 做最少但关键的工作**
- **流程稳定，不依赖 AI 记住太多底层协议**

---

## 2. 已确认的设计原则

### 2.1 AI 只负责产出和申报，不负责底层交付细节

AI 负责：

- 理解任务
- 生成代码或文件
- 明确声明哪些结果是最终交付物
- 填写必要的提交说明、摘要、标题

系统负责：

- 准备工作区
- 管理 Git 分支 / PR
- 管理文件上传
- 决定放 Git、S3 还是本地存储
- 注册 Artifact
- 管理返工后的版本关系

结论：

**AI 不直接承担存储后端管理、仓库生命周期管理、上传生命周期管理。**

### 2.2 所有交付物统一进入 Artifact 流程

不再把“代码交付”和“文件交付”设计成两条完全不同的业务链。

统一原则：

- 代码类结果最终也是 Artifact
- 文档、PPT、图片等非代码结果也是 Artifact
- 区别只在于 `kind` 不同、`content_ref` 不同、落地后端不同

这样返工、审阅、批准、拒绝、版本追踪可以共用一套逻辑。

### 2.3 AI 必须显式申报最终结果

系统不猜测“哪些文件算完成”，AI 需要通过工具或结构化结果显式说明：

- 这个任务最终产出了哪些文件
- 如果是代码任务，准备提交到哪个分支
- 如果需要 PR，PR 标题和描述是什么
- 如果是文件任务，这些文件的路径和摘要是什么

结论：

**交付以 AI 的显式申报为准，而不是以目录里是否出现文件为准。**

### 2.4 正式提交和上传动作由系统执行

代码类结果的正式交付动作包括：

- `git add`
- `git commit`
- `git push`
- 创建 Pull Request

文件类结果的正式交付动作包括：

- 文件校验
- 上传到目标存储
- 返回稳定引用
- 注册 Artifact

这些动作由系统执行，不由 AI 直接操作外部系统。

结论：

**AI 声明“我要提交什么”，系统负责“真正提交”。**

### 2.5 返工走版本化，不走覆盖

返工后不能直接覆盖旧文件或旧 Artifact。

需要保留：

- 哪一版被提交
- 哪一版被拒绝
- 哪一版替代了上一版
- 当前生效的是哪一版

结论：

**Artifact 必须支持版本化和 supersede 关系。**

## 3. 工作区与代码仓库方案

### 3.1 代码任务使用系统准备的 worktree

当前认可的方向：

- 仓库主 clone 由系统维护
- 每个代码任务创建独立的 `git worktree`
- AI 只在分配给该任务的 worktree 中工作

这样做的原因：

- 比每个任务重新 clone 更高效
- 比让 AI 自己决定 clone 位置、branch 状态更稳定
- 并行任务天然隔离
- 返工可以延续同一个任务的工作区语义

### 3.2 非代码任务使用普通 workspace

如果任务不需要 Git 仓库，例如纯文案、PPT、图片生成，则系统准备普通目录工作区。

建议目录结构：

```text
{workspace}/
├── source/
├── artifacts/
├── scratch/
└── manifest/
```

含义：

- `source/`：源材料、输入文件、参考文件
- `artifacts/`：最终准备交付的结果
- `scratch/`：临时中间文件
- `manifest/`：系统元数据

### 3.3 AI 不直接管理仓库初始化和底层 Git 生命周期

已经确认的原则是：

- 不要求 AI 自己 clone 仓库
- 不要求 AI 自己创建和维护底层工作区
- 不要求 AI 自己决定 push 权限和远端细节

AI 的工作边界应当是：

- 在给定工作区里修改文件
- 在交付时声明分支名、提交说明和 PR 文案

## 4. AI 如何告诉系统它完成了什么

### 4.1 需要“交付声明”而不是自由文本汇报

如果只让 AI 在聊天里说：

- 我改好了代码
- 我写好了文档
- 我做完了 PPT

系统无法稳定解析出真正要交付的对象。

因此需要让 AI 在结束任务时提交**结构化交付声明**。

### 4.2 代码类交付声明

代码类任务中，AI 需要告诉系统：

- 当前工作区路径
- 目标分支名
- 提交说明
- PR 标题
- PR 描述
- 本次期望提交的文件范围
- 改动摘要

这类声明随后由系统校验并执行真正的 Git 提交与 PR 创建。

### 4.3 文件类交付声明

文档、PPT、图片、视频等非代码任务中，AI 需要告诉系统：

- 当前工作区路径
- 交付物类型
- 结果标题
- 摘要
- 最终文件列表
- 可选的元数据

这类声明随后由系统校验并执行上传、归档和 Artifact 注册。

### 4.4 AI 的最终结果报告应引用系统返回值

系统执行完正式交付动作后，会返回稳定引用，例如：

- `artifact_id`
- `branch`
- `pr_url`
- `content_ref`

AI 最终对任务的完成报告，应引用这些系统返回值，而不是继续在自由文本里重复底层路径和 URL。

## 5. Artifact 模型需要补强的部分

### 5.1 Artifact 的定位

Artifact 不是“附件”这么简单，它是流程里真正被审阅、批准、拒绝、返工的对象。

Task 是流程骨架。

Artifact 是交付载体。

### 5.2 建议的关键字段

建议在 v3 的 Artifact 模型中至少具备以下能力：

```text
Artifact
├── id
├── task_id
├── kind
├── title
├── summary
├── producer_id
├── status
├── version
├── supersedes_artifact_id
├── content_refs
├── metadata
├── created_at
└── updated_at
```

重点字段含义：

- `kind`：代码改动、文档、PPT、图片等
- `status`：draft / submitted / approved / rejected / superseded
- `version`：当前是第几版
- `supersedes_artifact_id`：替代哪一版
- `content_refs`：真实内容存放位置
- `metadata`：附加元数据，如文件大小、媒体类型、导出参数

### 5.3 content_ref 继续保持抽象

当前结论不是把 Artifact 绑定到 Git 或 S3，而是保持引用抽象：

```json
[
  {"kind": "git_pr", "uri": "https://github.com/org/repo/pull/123"},
  {"kind": "s3", "uri": "s3://bucket/task-001/deck-v2.pptx"}
]
```

这样未来即使接入：

- S3 / OSS
- GitHub / GitLab
- Notion / 飞书 / 内部文档系统

都不需要推翻 Artifact 模型。

## 6. Git、S3、本地存储怎么分工

### 6.1 当前认可的存储原则

不是所有文件都进 Git，也不是所有文件都进 S3。

推荐分工：

- **代码类产物**：优先 Git
- **小型文本类产物**：通常也可进 Git
- **大文件、二进制、图片、PPT、视频**：优先对象存储，例如 S3
- **当前外部存储未准备好时**：可先放本地 workspace 或本地归档目录，但 Artifact 引用层不要写死

### 6.2 为什么代码更适合 Git

代码类结果需要：

- diff
- review
- branch
- pull request
- 历史追踪

这些都是 Git 的强项。

### 6.3 为什么二进制产物更适合对象存储

PPT、图片、视频、压缩包等文件如果长期进 Git，会出现：

- 仓库膨胀
- 二进制 diff 不友好
- clone/fetch 成本越来越高
- 清理历史困难

因此这类文件更适合放对象存储。

### 6.4 存储选择由系统策略决定

已确认不让 AI 决定存 Git 还是 S3。

系统需要一层策略，根据以下因素决定：

- Artifact 类型
- 文件大小
- 是否二进制
- 是否需要 PR review
- 当前环境是否配置了对象存储

这层策略未来可以抽象成 `ArtifactPublishPolicy` 或类似组件。

## 7. 代码任务的交付链路

代码类任务的有效流程可以整理为：

```text
系统准备 worktree
-> AI 在 worktree 中改代码
-> AI 提交代码类交付声明
-> 系统校验改动范围和提交参数
-> 系统执行 commit / push / create PR
-> 系统注册 Artifact(kind=code_change)
-> 进入 review / gate
```

这里已经确认的边界有两条：

- AI 可以写 commit message、PR title、PR body
- 系统执行正式的 Git 外部动作

这样既保留 AI 对结果语义的表达能力，又把易错的权限和状态操作留在系统端。

---

## 8. 文件任务的交付链路

文档、PPT、图片、视频等非代码任务的有效流程可以整理为：

```text
系统准备普通 workspace
-> AI 生成文件并放入约定目录
-> AI 提交文件类交付声明
-> 系统校验文件存在、路径合法、类型合法
-> 系统决定存储后端并执行上传/归档
-> 系统注册 Artifact
-> 进入 review / gate
```

这里的关键点是：

- AI 报告“哪些文件是最终结果”
- 系统负责“上传之后变成什么引用”

这避免了让 AI 自己处理上传 URL、对象键命名、去重、版本归档等问题。

## 9. 返工、重提与版本关系

### 9.1 为什么不能覆盖

如果返工时直接覆盖旧文件，会失去：

- 审阅针对的是哪一版
- 被拒绝的是哪一版
- 本轮改动相对上一版改了什么

这会直接破坏可追溯性。

### 9.2 确认的版本化原则

每次返工都生成新版本 Artifact：

```text
v1 submitted
v2 rejected
v3 submitted
v4 approved
```

版本之间需要能表达：

- 当前版本号
- 是否替代上一版
- 当前生效版本是谁

### 9.3 审阅意见必须和具体版本绑定

审阅与 Gate 判断不能只挂在 Task 上，至少要能指向：

- 某个 Artifact
- 甚至某个 Artifact version

否则返工后很容易丢失语义上下文。

## 10. 对 AI 的执行契约

这部分需要最终沉淀到 prompt 或工具说明里。

### 10.1 代码任务

AI 需要遵守：

- 只在系统分配的 worktree 中工作
- 不直接执行正式的 push / PR 创建
- 完成后显式提交代码交付声明
- 最终汇报引用系统回填的分支和 PR 结果

### 10.2 文件任务

AI 需要遵守：

- 将最终产物放在约定交付目录
- 临时文件与最终产物分离
- 不直接管理外部上传
- 完成后显式提交文件交付声明
- 最终汇报引用系统回填的 Artifact 和内容引用

### 10.3 统一边界

AI 需要知道的是业务语义：

- 我完成了什么
- 这些文件是什么
- 它们对应的标题、摘要和说明是什么

AI 不需要深度知道的是底层平台细节：

- S3 key 怎么命名
- PR API 怎么调
- Git 远端权限怎么配
- 归档路径如何规划

## 11. 当前仍待继续定稿的开放点

以下不是废弃方案，而是还需要继续明确的有效设计问题。

### 11.1 交付声明是做成工具调用，还是允许结构化文本回填

目前更倾向工具调用，因为：

- 校验更稳定
- 结果更结构化
- 更利于系统自动执行后续动作

但最终仍要决定：

- 是否完全禁止自由文本回填
- 是否允许在某些环境下退化为结构化 JSON 文本

### 11.2 文件类 Artifact 的本地落点是否进 Git

当前方向是：

- 小型文本可以进 Git
- 二进制优先不进 Git

但仍需继续明确：

- 纯 Markdown 报告默认进 Git 还是走统一 ArtifactStore
- 单仓库项目与多项目空间下，文本交付物是否共用同一规则

### 11.3 是否需要单独的 ArtifactStore 抽象

现在在概念上已经有这个方向：

- Git 类落地
- 对象存储类落地
- 本地文件类落地

后续需要决定：

- 是直接在现有服务里做分支判断
- 还是抽象成独立的 `ArtifactStore` / `ArtifactPublishPolicy`

### 11.4 代码任务的提交边界由谁校验

当前认可“系统执行正式提交”，但仍需明确：

- 改动文件范围是否由 Task 配置约束
- 是否允许 AI 提交工作区中的全部变更
- 多 agent 并行时如何限制跨任务污染

### 11.5 文件类任务的版本命名规则

虽然 Artifact 会版本化，但真实文件名、对象键、目录布局是否也要版本化，还需要明确：

- 是否强制 `v1 / v2 / v3`
- 是否保留稳定别名指向 latest
- 审批通过后是否生成正式发布路径

---

## 12. 目前收敛出的结论

到当前为止，围绕“AI 做最少工作但流程稳定运行”，已经形成的结论可以压缩成下面几条：

- AI 负责生成结果和显式申报结果
- 系统负责准备 workspace、正式提交、正式上传和 Artifact 注册
- 代码任务优先使用 `git worktree`
- 代码结果优先落 Git，二进制和大文件优先落对象存储
- 所有结果统一收敛到 Artifact
- 返工采用版本化与 supersede 关系，而不是覆盖
- AI 的最终汇报应引用系统回填的稳定结果，而不是自由拼接底层路径

这套设计与 v3 的核心原则一致：

**让 AI 负责语义判断，让系统负责确定性动作。**
