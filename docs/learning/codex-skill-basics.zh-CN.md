# Codex Skill 基础理解

> 整理时间：2026-03-30
> 文档类型：学习记录（基于 `skill-creator`，非项目规范）
> 研究主题：什么是 Skill

---

## 一句话定义

Skill 是给 Codex 使用的“专项能力包”。

它不是普通说明文档，而是一个模块化、自包含的文件夹，里面放着执行某类任务需要的工作流、规则、脚本和参考资料，让通用 Codex 在特定场景下表现得更专业、更稳定。

---

## 1. Skill 的本质

从 `skill-creator` 的定义看，Skill 更像是给另一个 Codex 准备的“上岗指南”：

- 告诉它这类任务什么时候该触发
- 告诉它这类任务应该怎么做
- 在必要时给它脚本、资料和模板

可以把它理解成：

`Skill = 面向 AI 的 SOP + 工具箱 + 资料导航`

它的目标不是讲概念，而是提高执行质量和稳定性。

---

## 2. Skill 主要提供什么

文档把 Skill 的价值概括为四类能力：

### 2.1 Specialized workflows

也就是特定任务的标准流程。

比如：

- 如何处理 PDF
- 如何查询某个内部数据源
- 如何完成某类固定的代码工作流

### 2.2 Tool integrations

也就是告诉 Codex 如何正确使用某些工具、API 或文件格式。

比如：

- 如何调用某个 CLI
- 如何处理特定格式文件
- 如何按约定调用外部系统

### 2.3 Domain expertise

也就是模型默认不知道、但任务里又很关键的领域知识。

比如：

- 公司内部约定
- 数据库 schema
- 业务术语
- 行业规则

### 2.4 Bundled resources

也就是可复用的配套资源。

比如：

- 脚本
- 参考文档
- 模板
- 素材文件

---

## 3. 一个 Skill 的最小结构

一个 Skill 至少是一个文件夹，并且必须包含 `SKILL.md`：

```text
skill-name/
├── SKILL.md
├── agents/
│   └── openai.yaml
├── scripts/
├── references/
└── assets/
```

其中：

- `SKILL.md` 是核心入口，必须存在
- `agents/openai.yaml` 是推荐项，主要服务 UI 展示和技能信息
- `scripts/` 用来放稳定、可复用、适合反复执行的脚本
- `references/` 用来放按需读取的参考材料
- `assets/` 用来放最终输出会直接用到的资源

---

## 4. `SKILL.md` 为什么最重要

`SKILL.md` 是 Skill 的中心。

它分成两部分：

### 4.1 Frontmatter

只要求两个字段：

- `name`
- `description`

其中最关键的是 `description`，因为它承担“触发条件”的作用。

也就是说，Codex 主要靠这里判断：

- 这个 Skill 是干什么的
- 用户的请求是否应该触发它

### 4.2 Body

正文不是用来重复写“什么时候用”，而是用来写：

- 具体操作说明
- 工作流约束
- 如何使用资源文件

简化理解就是：

- `description` 负责“该不该用”
- `body` 负责“用了以后怎么做”

---

## 5. Skill 怎么被加载

文档强调了一个很重要的原则：**渐进加载（Progressive Disclosure）**。

Skill 不会一开始把所有内容都塞进上下文，而是分 3 层：

### 5.1 第一层：元数据

始终可见，主要是：

- `name`
- `description`

这层用于触发判断。

### 5.2 第二层：`SKILL.md` 正文

只有 Skill 被触发后，Codex 才会读正文。

### 5.3 第三层：资源文件

只有在确实需要时，Codex 才会去读：

- `scripts/`
- `references/`
- `assets/`

这套设计的核心目标是：**省上下文，按需展开。**

---

## 6. 为什么 Skill 要尽量短

`skill-creator` 的一个核心原则是：**默认 Codex 已经很聪明。**

所以 Skill 不应该重复常识，而应该只补充：

- 模型原本不知道的内容
- 容易做错的流程
- 需要稳定复用的资源

因此一个好的 Skill 往往具备这些特点：

- 短
- 准
- 可触发
- 可执行
- 可按需展开

不是内容越多越好，而是越聚焦越好。

---

## 7. Skill 不是普通项目文档

文档明确指出，不要把 Skill 做成一堆给人看的说明书。

不建议额外堆这些文件：

- `README.md`
- `CHANGELOG.md`
- `INSTALLATION_GUIDE.md`
- `QUICK_REFERENCE.md`

原因不是这些文件没价值，而是它们通常会让 Skill 变重、变散、变得不利于 AI 执行。

Skill 的目标很明确：

> 帮另一个 Codex 更高效地完成任务，而不是完整记录人工阅读材料。

---

## 8. Skill 和普通 Prompt / Script 的区别

### 8.1 和普通 Prompt 的区别

普通 Prompt 通常只是一段话。

Skill 则是一个完整的能力单元，除了说明，还能包含：

- 触发描述
- 工作流
- 脚本
- 参考资料
- 输出资源

### 8.2 和单个 Script 的区别

脚本只解决“怎么执行”。

Skill 还解决：

- 什么时候执行
- 为什么这样执行
- 遇到不同场景时怎么选路径

### 8.3 和插件的区别

插件更偏“新增工具能力”。

Skill 更偏“新增任务执行知识和流程”。

插件解决“能不能做”，Skill 解决“怎么更好地做”。

---

## 9. 当前理解下的结论

基于 `skill-creator`，可以把 Skill 总结为：

- 一个面向 Codex 的能力扩展包
- 以 `SKILL.md` 为核心
- 通过 `description` 决定触发
- 通过正文指导执行
- 通过 `scripts / references / assets` 提供可复用资源
- 通过渐进加载控制上下文成本

最关键的一点是：

**Skill 不是“写很多说明”，而是“把某类任务压缩成一个可复用、可触发、可执行的能力模块”。**

---

## 10. 后续可继续学习的方向

如果后面继续往下拆，可以重点看三个问题：

1. 一个好的 `SKILL.md` 应该怎么写
2. 什么场景适合做成 Skill，什么场景不适合
3. 如何把现有脚本、知识和流程沉淀成 Skill，而不是堆成文档
